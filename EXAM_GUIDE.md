# Конспект для защиты Flutter-приложения

Этот файл помогает объяснить код на экзамене. Разбор сделан по учебной части проекта: `pubspec.yaml`, папка `lib` и тест из `test`. Сгенерированные платформенные файлы Android/iOS/Windows/Linux/macOS обычно не защищают построчно: они создаются командой `flutter create` и нужны для сборки под разные платформы.

## Общая идея приложения

Приложение называется "Складской учет". Оно решает задачу из лабораторной:

- пользователь регистрируется или входит в систему;
- роли делят пользователей на администратора и обычного пользователя;
- администратор может добавлять товары;
- все пользователи могут смотреть каталог и карточки товаров;
- товар имеет статус "свободен" или "занят";
- QR-код хранит идентификатор товара;
- сканирование QR-кода выдает товар или возвращает его на склад;
- данные хранятся локально в SQLite.

Главная архитектура:

- `models` - классы данных `AppUser`, `Product`, `UserRole`;
- `services` - работа с SQLite через `DatabaseService`;
- `state` - бизнес-логика и состояние через `WarehouseState`;
- `screens` - экраны интерфейса;
- `widgets` - переиспользуемые виджеты.

## pubspec.yaml

- `name: sklad_app` - техническое имя Flutter-проекта.
- `description` - краткое описание проекта.
- `publish_to: 'none'` - запрещает случайную публикацию пакета на pub.dev.
- `version: 1.0.0+1` - версия приложения и номер сборки.
- `environment sdk: ^3.11.1` - минимальная версия Dart SDK, с которой работает проект.
- `flutter sdk: flutter` - подключает Flutter SDK.
- `cupertino_icons` - стандартные иконки в стиле iOS, добавлены Flutter-шаблоном.
- `provider` - библиотека управления состоянием.
- `sqflite` - библиотека для локальной SQLite-базы.
- `path` - библиотека для корректной сборки пути к файлу базы.
- `qr_flutter` - генерация QR-кодов в интерфейсе.
- `mobile_scanner` - сканирование QR-кодов через камеру.
- `flutter_test` - инструменты для тестов.
- `flutter_lints` - правила анализа кода.
- `uses-material-design: true` - включает Material Icons.

## Как объяснить поток работы приложения

1. Пользователь запускает приложение.
2. `main.dart` создает `WarehouseState` и вызывает `initialize`.
3. `DatabaseService` открывает SQLite-базу и при первом запуске создает таблицы.
4. Если пользователь не вошел, `AppRoot` показывает `LoginScreen`.
5. При входе `WarehouseState.login` проверяет email, пароль и ищет пользователя в базе.
6. После входа `AppRoot` показывает `ProductListScreen`.
7. Каталог берет товары из `WarehouseState.products`.
8. Администратор видит кнопку добавления товара.
9. При добавлении товара `WarehouseState.addProduct` проверяет права и валидирует данные.
10. Карточка товара показывает QR-код `warehouse-product:ID`.
11. Сканер считывает QR-код и вызывает `WarehouseState.applyQrCode`.
12. Если товар свободен, он становится занятым.
13. Если товар занят, он возвращается на склад.
14. База обновляется, а интерфейс перерисовывается через `notifyListeners`.

## Частые вопросы на защите

**Почему используется Provider?**

Чтобы хранить состояние приложения в одном месте и автоматически обновлять экраны после изменения данных.

**Почему SQLite?**

Потому что по заданию нужно локальное хранение пользователей, товаров и информации о выдаче товара.

**Где разграничение прав?**

В `UserRole.canCreateProducts` и в проверке `WarehouseState.addProduct`. Добавлять товары может только администратор.

**Что хранится в QR-коде?**

Строка `warehouse-product:ID`, где `ID` - идентификатор товара в базе.

**Как работает выдача и возврат?**

Метод `applyQrCode` находит товар по id. Если он свободен, записывает текущего пользователя и дату. Если занят, очищает эти поля и делает товар свободным.

**Почему UI не работает с базой напрямую?**

Так код разделен на слои: экран отвечает за внешний вид, `WarehouseState` за логику, `DatabaseService` за хранение данных.
